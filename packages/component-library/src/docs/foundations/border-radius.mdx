Border radius decisions made ad hoc produce interfaces where similar elements have subtly different shapes, making the UI feel unpolished without an obvious cause.

Meteor provides two layers of border radius tokens: a named scale for general use, and semantic tokens that encode the correct radius for specific component types. Always prefer a semantic token when one exists for your use case. Fall back to the scale when no semantic token applies.

## Semantic tokens

Semantic tokens encode the right radius for specific element types. Use these before reaching for a scale step.

| Token                      | Value | Use for                              |
| -------------------------- | ----- | ------------------------------------ |
| `--border-radius-card`     | 8px   | Cards and container surfaces         |
| `--border-radius-button`   | 4px   | Buttons and button-like controls     |
| `--border-radius-checkbox` | 4px   | Checkboxes and similar input toggles |

## Scale tokens

Use scale tokens when no semantic token fits your element.

| Token                   | Value  |
| ----------------------- | ------ |
| `--border-radius-none`  | 0px    |
| `--border-radius-2xs`   | 2px    |
| `--border-radius-xs`    | 4px    |
| `--border-radius-s`     | 6px    |
| `--border-radius-m`     | 8px    |
| `--border-radius-l`     | 12px   |
| `--border-radius-xl`    | 16px   |
| `--border-radius-2xl`   | 20px   |
| `--border-radius-3xl`   | 24px   |
| `--border-radius-4xl`   | 32px   |
| `--border-radius-round` | 9999px |

`--border-radius-round` produces a fully rounded shape (pill or circle) regardless of element size.
