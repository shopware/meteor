## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline code padding trims, compact status indicator offsets |
| sm | 8px | Small chip gaps, list spacing, mobile tab gutters |
| md | 16px | Default internal spacing inside cards and between stacked preview sections |
| lg | 24px | Card padding, shell gaps, action-row spacing |
| xl | 32px | Page padding and major shell inset on desktop |
| 2xl | 48px | Major separation between top-level workspace groups when additional vertical rhythm is needed |
| 3xl | 64px | Reserved for page-level whitespace only; do not use inside cards |

Spacing exceptions are not allowed in this phase. Remap any legacy 12px or 20px spacing uses to the declared tokens above. All tappable controls must keep a 44px minimum height, treated as a control-height requirement rather than part of the spacing scale.

## Interaction Contract

### Layout And Placement

First-read hierarchy: before the first run, the preview card empty state and `Run preview` button are the primary focal point; after a run, the rendered preview surface becomes the primary focal point and diagnostics remain secondary support content below it.

| Area | Contract |
|------|----------|
| Workspace shell | Preserve the Phase 2 desktop split: guidance/navigation left, editor plus preview right. Do not introduce a third column in Phase 3. |
| Preview card position | Keep the preview card directly below the editor card in the right-hand pane on desktop and below the editor within the `Workspace` mobile section. |
| Preview card header | Left side: `Preview` heading and a single run-status pill. Right side: action group with `Run preview` as the primary button and `Restart preview` as the secondary button. |
| Preview surface | Successful runs render inside a neutral framed lesson-app surface with 18px radius, 1px border, white background, and a minimum height of 240px. Do not add fake-admin chrome, sidebars, or Shopware-styled top bars in this phase. |
| Diagnostics placement | Render a dedicated diagnostics panel inside the same preview card, directly below the preview surface, separated by 16px. Keep it visible in every state so layout does not jump between runs. |
| Mobile controls | On screens below the existing desktop breakpoint, wrap the preview header onto two rows and make the action buttons span the full available width before the diagnostics panel. |

### State Model

| State | Visual Treatment | Interaction Rule |
|-------|------------------|------------------|
| Ready to run | Empty preview surface with the empty-state heading/body and a muted neutral diagnostics placeholder | `Run preview` enabled. `Restart preview` hidden until a session has been started at least once. |
| Running | Accent status pill labeled `Running`; preview surface keeps its frame and shows a non-blocking loading indicator | Disable `Run preview` and `Restart preview` until the run settles. Keep the editor shell mounted. |
| Success | Render the isolated lesson app surface; diagnostics panel shows a neutral success message instead of disappearing | `Restart preview` becomes visible and enabled. |
| Draft changed after success | Keep the last successful preview visible and overlay or place an accent pill reading `Changes not run yet` in the preview header | Do not auto-run. The preview stays stale until the learner explicitly selects `Run preview`. |
| Compile error | Replace the preview surface contents with an error-state panel; do not continue showing the last successful output | Diagnostics panel opens in error styling and labels the failure as `Compile error`. |
| Runtime error | Replace the preview surface contents with an error-state panel; do not continue showing stale success output | Diagnostics panel opens in error styling and labels the failure as `Runtime error`. |

### Diagnostics Contract

| Element | Contract |
|---------|----------|
| Panel structure | Use a single stacked panel, not tabs. First row: error type or neutral status. Second row: primary message. Third row: monospace details block for stack traces or compiler output. |
| Error styling | Use the destructive red only for error headings, left-border accents, and critical chips. Keep the panel background light so it still reads as part of the tutorial shell. |
| Success styling | Show `No issues from the last run.` in muted body copy after a successful execution. Do not use success green in this phase. |
| Empty diagnostics | Show `Diagnostics will appear here after you run the lesson.` before the first run. |

### Action Semantics

| Action | Contract |
|--------|----------|
| Run preview | Always executes the current learner draft, never the solution panel, and always starts from the current draft content at the moment the button is pressed. |
| Restart preview | Resets only the isolated preview runtime session. It must preserve the learner draft and must not require a confirmation dialog. |
| Restore starter | Keeps the existing destructive confirmation dialog from Phase 2. It is separate from preview restart and must not be visually grouped as if it affects runtime only. |
